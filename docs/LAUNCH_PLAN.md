# План запуска VPN-стартапа и масштабирование

## Фазы запуска

### Фаза 1: MVP (4–6 недель)

1. **Инфраструктура**
   - 1 VPS для Backend (API + PostgreSQL можно на одном сервере или managed DB).
   - 1 Redis (managed или на отдельном VPS).
   - 2–3 VPS для VPN (разные регионы: EU, US, Asia). На каждом — Xray-core + VLESS (Reality или TLS).

2. **Backend**
   - Деплой API (Docker или бинарник), миграции БД (schema.sql).
   - Настройка домена и HTTPS (Let's Encrypt или облачный LB).

3. **Мобильное приложение**
   - Flutter: экран логина/регистрации, главный экран с кнопкой Connect, выбор сервера, отображение статуса и подписки.
   - Интеграция с API (login, config, servers, subscribe).
   - Встроенный VPN-клиент (например, v2ray-core через плагин или платформенный VPN API с конфигом VLESS).

4. **Подписки**
   - Подключение Apple IAP и Google Play Billing (серверная верификация рецептов).
   - Backend: POST /api/subscribe с external_id после успешной оплаты.

5. **Админка**
   - Базовое управление серверами и пользователями, просмотр подписок.

### Фаза 2: Рост (2–3 месяца)

- Добавление регионов (новые VPS, регистрация в админке).
- Усиление анти-абьюза: лимит устройств, мониторинг трафика.
- Панель аналитики: DAU, конверсия в подписку, отток.

### Фаза 3: Масштабирование

- Backend: несколько инстансов за load balancer; PostgreSQL — managed (например, AWS RDS, DigitalOcean Managed DB).
- Redis: кластер или managed Redis для сессий и rate limit.
- VPN-ноды: автоматическое добавление через админку/скрипты; health check и исключение мёртвых нод из выдачи.
- Опционально: Kubernetes для API и воркеров; VPN-ноды остаются на VPS с Xray.

## Рекомендации по масштабированию

- **База данных**: индексы на user_id, expires_at, (user_id, device_id); при росте — партиционирование traffic_logs по дате.
- **Конфиги**: генерация VLESS на лету (как сейчас) масштабируется; при необходимости кэшировать в Redis по user_id с TTL 5–10 минут.
- **Серверы**: метрики нагрузки (подключения, CPU) — опционально экспорт в Prometheus/Redis; в GET /servers отдавать приоритет по наименьшей нагрузке.
- **Геораспределение**: API и БД ближе к пользователям (один регион на старте достаточно); VPN-ноды — в целевых странах для низкого пинга.
